# System Model

## Scope and Objective
Conclusion
- The analysis focuses on one core issue: insecure randomness in session token generation.
- The security objective is to protect token unpredictability and authentication state integrity.

Evidence
- Token generation location: `apk-decompile_code/sources/com/example/mastg_test0016/Login.java` lines 183-188.
- Token persistence location: `Login.java` lines 174-176.

## Main Components
Conclusion
- The system includes three main app components and two local data stores.

Evidence
- Activities in app structure: `apk-decompile_code/resources/AndroidManifest.xml` lines 28-37 show `MainActivity`, `Login`, `Profile`.
- Credential write path: `apk-decompile_code/sources/com/example/mastg_test0016/MainActivity.java` includes `saveCredentialsToFile`.
- Session lifecycle path: `Login.java` lines 174-176 and `apk-decompile_code/sources/com/example/mastg_test0016/Profile.java` lines 50-52.

## Assets
Conclusion
- The key assets are token unpredictability and authentication state integrity.
- Local credentials are contextual assets but not the selected core vulnerability.

Evidence
- Token is stored as `sessionToken` in `SharedPreferences`: `Login.java` lines 174-176.
- Logout clears token state: `Profile.java` lines 50-52.

## Data Flows
Conclusion
- The security-relevant flow is end to end from user input to session state.
- The core path is `Random -> Token -> SharedPreferences -> Session`.

Evidence
- Registration and local write: `MainActivity.saveCredentialsToFile`.
- Credential verification: `Login.checkCredentials`.
- Session creation and token generation: `Login.java` lines 174-176 and 183-188.
- Session endpoint behavior: `Profile` flow and `clearSession` method.

## Trust Boundaries
Conclusion
- Trust boundaries exist at input handling, authentication decision, token generation, and token persistence.
- The weak random generator is placed at a security decision boundary, not in display-only logic.

Evidence
- Auth token generated by `Random`: `Login.java` line 184.
- Contrast path with UI-only random value: `MainActivity.java` lines 17-20.

## Model Justification
Conclusion
- This model supports selecting F1 as the core vulnerability because the weak source directly affects persisted authentication state.

Evidence
- `generateSessionToken` is called from session creation and stored for session behavior: `Login.java` lines 174-176 and 183-188.
