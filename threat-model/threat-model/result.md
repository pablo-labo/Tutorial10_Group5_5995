# Threat Model (Core Result)

## 1) Threat statement linked to chosen vulnerability
Chosen vulnerability: session token generated by `java.util.Random` in `Login.generateSessionToken()` (`Login.java` lines 183-188).

Threat statement:
A deterministic non-cryptographic PRNG is used for security-sensitive token generation. Under realistic observation conditions, attackers may reduce candidate space and attempt session impersonation.

## 2) Attacker goals
Primary goal:
- Obtain or reproduce a valid session token to impersonate authenticated state.

Secondary goal:
- Reduce uncertainty in token space enough to make candidate testing practical.

## 3) Attacker capabilities (realistic, explicit)
- Can estimate victim login timing within bounded error.
- Can obtain partial token signal (prefix/sample) in practical environments (debug leakage, instrumentation, rooted-device or test-device contexts).
- Can run offline candidate generation using known PRNG behavior.

## 4) Attacker constraints (explicit)
- No assumption of remote code execution.
- No assumption of backend compromise.
- No claim of guaranteed success in all environments.

## 5) Preconditions mapped to exploit path
P1: Timing knowledge is available.
P2: Some token information is observable.
P3: Candidate validation channel exists.

If P1+P2 hold, predictability risk is materially higher than with CSPRNG-based token generation.

## 6) Why threat is realistic and not exaggerated
- We state bounded assumptions, not omniscient attacker power.
- We separate “design weakness exists” from “practical success probability under environment constraints”.
- We avoid out-of-scope vulnerability classes.

## 7) Security reasoning
`java.util.Random` is deterministic and not intended for security token generation. Authentication tokens should come from a CSPRNG source (e.g., `SecureRandom`) to resist prediction attacks.
