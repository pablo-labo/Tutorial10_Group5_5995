# B Speaker Script (>= 40 seconds)

In our system model, the app has three key components: `MainActivity`, `Login`, and `Profile`, with session state stored locally in `SharedPreferences` as `sessionToken`. The key security asset is not just credentials, but the integrity of this session token because it represents authenticated state. In the threat model, we assume a realistic attacker who can estimate login timing and observe partial token output in debug or compromised-device contexts. The critical point is that token generation uses `java.util.Random`, which is deterministic and not cryptographically secure for authentication tokens. So our risk claim is that session predictability can enable impersonation attempts under bounded conditions, and this is why this vulnerability should be prioritized as the core randomness/crypto finding.
