# Predictable Session Token Generation in an Android APK Due to `java.util.Random`

## Abstract
We analyzed the provided Android APK and selected one vulnerability that clearly fits the Assignment 1 randomness and cryptography scope. After successful credential verification, the app generates a value named `sessionToken` using `java.util.Random` and stores it in `SharedPreferences`. Because session tokens are security-sensitive authentication state and require unpredictability, a deterministic non-cryptographic PRNG is unsuitable for this role. Our evidence supports a bounded claim of weakened session-token unpredictability under realistic local-analysis conditions rather than a guaranteed authentication-bypass claim. The primary mitigation is to replace `java.util.Random` with `SecureRandom` in the token-generation path and preserve explicit token lifecycle controls.

## 1. Introduction
The provided APK is a small Android app with three main activities: `MainActivity`, `Login`, and `Profile` ([`AndroidManifest.xml`](/Users/ruben/Documents/Git%20docs/26s1/Tutorial10_Group5_5995/apk-decompile_code/resources/AndroidManifest.xml#L28)). In the observed flow, `MainActivity` stores user credentials locally, `Login` checks those credentials, and a successful login creates session-related state before opening `Profile` ([`MainActivity.java`](/Users/ruben/Documents/Git%20docs/26s1/Tutorial10_Group5_5995/apk-decompile_code/sources/com/example/mastg_test0016/MainActivity.java#L54), [`Login.java`](/Users/ruben/Documents/Git%20docs/26s1/Tutorial10_Group5_5995/apk-decompile_code/sources/com/example/mastg_test0016/Login.java#L52), [`Profile.java`](/Users/ruben/Documents/Git%20docs/26s1/Tutorial10_Group5_5995/apk-decompile_code/sources/com/example/mastg_test0016/Profile.java#L23)).

We selected one issue only: weak generation of the authentication-related `sessionToken` in `Login.generateSessionToken()`. This choice is directly within the assignment scope because it is a misuse of randomness in a security-sensitive path, not merely a cosmetic or UI-only use of randomness. Our goal is to present one strong, defensible finding with clear code evidence, a realistic attacker model, a concrete attack scenario, and a mitigation that directly addresses the demonstrated weakness.

## 2. System and Threat Model
The relevant system components are `MainActivity`, which writes credentials to `credentials.txt`; `Login`, which validates credentials and creates session state; and `Profile`, which represents the post-login state and clears session data on logout ([`MainActivity.java`](/Users/ruben/Documents/Git%20docs/26s1/Tutorial10_Group5_5995/apk-decompile_code/sources/com/example/mastg_test0016/MainActivity.java#L62), [`Login.java`](/Users/ruben/Documents/Git%20docs/26s1/Tutorial10_Group5_5995/apk-decompile_code/sources/com/example/mastg_test0016/Login.java#L174), [`Profile.java`](/Users/ruben/Documents/Git%20docs/26s1/Tutorial10_Group5_5995/apk-decompile_code/sources/com/example/mastg_test0016/Profile.java#L50)). The primary protected asset in our selected path is the unpredictability and integrity of authentication-related session state represented by `sessionToken`. The key data flow is: successful credential verification in `Login` -> `createSession()` generates `sessionToken` -> the token is stored in `SharedPreferences("SessionPrefs")` -> later code can retrieve or clear that state through `getSessionToken()` and `clearSession()` ([`Login.java`](/Users/ruben/Documents/Git%20docs/26s1/Tutorial10_Group5_5995/apk-decompile_code/sources/com/example/mastg_test0016/Login.java#L52), [`Login.java`](/Users/ruben/Documents/Git%20docs/26s1/Tutorial10_Group5_5995/apk-decompile_code/sources/com/example/mastg_test0016/Login.java#L174), [`Login.java`](/Users/ruben/Documents/Git%20docs/26s1/Tutorial10_Group5_5995/apk-decompile_code/sources/com/example/mastg_test0016/Login.java#L179), [`Profile.java`](/Users/ruben/Documents/Git%20docs/26s1/Tutorial10_Group5_5995/apk-decompile_code/sources/com/example/mastg_test0016/Profile.java#L50)).

The most important trust boundary in this vulnerability is crossed when the app generates a token inside the `Login` activity and persists it into `SharedPreferences("SessionPrefs")` under the key `sessionToken` ([`Login.java`](/Users/ruben/Documents/Git%20docs/26s1/Tutorial10_Group5_5995/apk-decompile_code/sources/com/example/mastg_test0016/Login.java#L26), [`Login.java`](/Users/ruben/Documents/Git%20docs/26s1/Tutorial10_Group5_5995/apk-decompile_code/sources/com/example/mastg_test0016/Login.java#L44), [`Login.java`](/Users/ruben/Documents/Git%20docs/26s1/Tutorial10_Group5_5995/apk-decompile_code/sources/com/example/mastg_test0016/Login.java#L174)). At that point, a value that should serve as trustworthy authentication-related state moves from internal generation logic into persistent app state. If the generation step is weak, the weakness propagates into the stored session state that the rest of the app may treat as meaningful.

Our attacker model is intentionally bounded. We assume a local-analysis attacker using an emulator, rooted device, or instrumentation/debug environment who can estimate when a victim logs in and may be able to inspect or test locally stored token state. The attacker's realistic goal is not full remote takeover, but to reduce the uncertainty of a security-sensitive token enough to make offline reproduction or local comparison materially easier than intended. We do not assume backend compromise, remote code execution, or a proven downstream server-side token-validation sink in the current APK. This model is realistic for an APK reverse-engineering assignment and avoids exaggerating the impact beyond what the repository evidence supports.

## 3. Vulnerability Discovery and Explanation
We decompiled the APK and searched for candidate randomness usage related to security-sensitive values such as `random`, `token`, `session`, and `login`. Static analysis revealed two visible uses of randomness. The first is `MainActivity.randomNumberGenerator()`, which returns a random integer for a UI-adjacent code path and has no clear authentication role ([`MainActivity.java`](/Users/ruben/Documents/Git%20docs/26s1/Tutorial10_Group5_5995/apk-decompile_code/sources/com/example/mastg_test0016/MainActivity.java#L17)). The second is `Login.generateSessionToken()`, which is invoked by `createSession()` immediately after successful credential verification ([`Login.java`](/Users/ruben/Documents/Git%20docs/26s1/Tutorial10_Group5_5995/apk-decompile_code/sources/com/example/mastg_test0016/Login.java#L52), [`Login.java`](/Users/ruben/Documents/Git%20docs/26s1/Tutorial10_Group5_5995/apk-decompile_code/sources/com/example/mastg_test0016/Login.java#L174)).

The selected path is clearly stronger and more security-relevant. `createSession()` stores a value under the key `sessionToken` in `SharedPreferences`, `getSessionToken()` retrieves it, and `Profile.clearSession()` removes it on logout ([`Login.java`](/Users/ruben/Documents/Git%20docs/26s1/Tutorial10_Group5_5995/apk-decompile_code/sources/com/example/mastg_test0016/Login.java#L174), [`Login.java`](/Users/ruben/Documents/Git%20docs/26s1/Tutorial10_Group5_5995/apk-decompile_code/sources/com/example/mastg_test0016/Login.java#L179), [`Profile.java`](/Users/ruben/Documents/Git%20docs/26s1/Tutorial10_Group5_5995/apk-decompile_code/sources/com/example/mastg_test0016/Profile.java#L50)). This lifecycle shows that the value is intended to represent authentication-related session state rather than a harmless display value.

The generation logic itself is the core vulnerability. `generateSessionToken()` constructs a 16-character token from an alphanumeric alphabet, but the randomness source is `new Random()` and the token characters are chosen via repeated `nextInt(62)` calls ([`Login.java`](/Users/ruben/Documents/Git%20docs/26s1/Tutorial10_Group5_5995/apk-decompile_code/sources/com/example/mastg_test0016/Login.java#L183)). `java.util.Random` is a deterministic PRNG, not a cryptographically secure random generator. That distinction matters because a session token should be unpredictable to anyone except the legitimate holder. Even though the nominal output space looks large, security depends on the unpredictability of the generator in realistic conditions, not only on the token alphabet or length.

Therefore, the weakness is not “the app uses randomness somewhere.” The weakness is that the app uses a predictable non-CSPRNG to generate a value explicitly named and handled as `sessionToken` in the authentication flow.

## 4. Attack Scenario and Impact
In the observed login path, successful credential verification in `Login.onClick()` triggers `createSession()`, which stores a freshly generated `sessionToken`, and then opens `Profile` ([`Login.java`](/Users/ruben/Documents/Git%20docs/26s1/Tutorial10_Group5_5995/apk-decompile_code/sources/com/example/mastg_test0016/Login.java#L52)). A concrete bounded scenario is as follows. First, the attacker observes or closely estimates the victim's login time in an emulator, rooted-device, or instrumented environment. Second, the attacker reproduces the exact `generateSessionToken()` logic offline, using the same `Random`-based algorithm and candidate seed windows around the observed login time. Third, if the attacker can inspect local `SharedPreferences` state or test token-dependent behavior in that environment, the candidate outputs can be compared against the stored token or against behavior that depends on it. The attack therefore does not require us to prove a remote exploit chain; it only requires a realistic path by which deterministic token generation makes local prediction or reproduction more credible than it should be.

This scenario is concrete enough to justify a real security claim, but it also has an important evidentiary limit. The repository shows token creation, storage, retrieval, and clearing more clearly than downstream token-validation enforcement. For that reason, the strongest defensible impact is weakened unpredictability of authentication-related session state, not a guaranteed authentication bypass, session hijack, or remote takeover.

This bounded impact claim is still significant. Session tokens are security-labelled values. If they can be predicted or reproduced more easily than intended because they are derived from a non-cryptographic PRNG, then the design fails a core security property required for authentication-related state.

## 5. Fix and Mitigation
The primary mitigation is to replace `java.util.Random` with `java.security.SecureRandom` in `Login.generateSessionToken()`. This directly addresses the demonstrated weakness because it removes the predictable PRNG from the exact path used to create `sessionToken`. A corrected implementation should continue to generate the token from a sufficiently large alphabet or, preferably, use secure random bytes encoded into a token representation with adequate entropy.

The key reason this fix works is that the attack path above depends on the determinism of `java.util.Random`: if an attacker can narrow the seed window, they can enumerate plausible outputs from the same generator. Replacing that generator with `SecureRandom` breaks that reasoning path because the token is no longer credibly enumerable from nearby timing assumptions in the same way. To strengthen the design further, the app should keep explicit token rotation on login and invalidation on logout, and it should make token validation explicit before granting access to protected flow. These extra controls are valuable because they improve overall session handling, but they should be described as strengthening measures rather than proof that the current APK already enforces token-based access correctly. Based on the current evidence, the strongest supported claim is that replacing `Random` with `SecureRandom` blocks the predictability path we identified.

We used APK reverse-engineering and static inspection to discover the vulnerability. The workflow included examining `AndroidManifest.xml`, reading the decompiled Java classes, searching for candidate terms such as `random`, `token`, `session`, and `login`, and collecting screenshot evidence from the selected code path. This process let us distinguish the non-selected UI-only random usage in `MainActivity` from the security-sensitive token-generation path in `Login`.

## 7. Conclusion
The provided APK generates an authentication-related `sessionToken` with `java.util.Random` immediately after successful login and stores it as session state. Because session tokens require unpredictability, this is an inappropriate use of a deterministic non-cryptographic PRNG in a security-sensitive context. Our strongest evidence-backed conclusion is a bounded predictability-risk claim under realistic local-analysis conditions. Replacing `Random` with `SecureRandom` is the direct and appropriate fix.
